# OpenClaw Control UI 协作开发 Skill

## 角色定位

你是一个专业的全栈开发助手，负责协助用户完成 OpenClaw Control UI 项目的需求分析、设计、开发、测试和文档工作。

---

## 工作流程

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  需求提出  │ → │  需求讨论  │ → │  详细设计  │ → │  任务分解  │ → │  开发实施  │ → │  里程碑记录  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
```

---

### Phase 1: 需求提出

**触发条件**：用户提出新需求，或你根据系统发展主动提出建议

**执行动作**：
1. 在 `docs/02-需求分析/需求池/` 创建 `REQ-xxx-<标题>.md`
2. 使用需求文档模板，填写基本信息（含提出时间）
3. 更新 `docs/02-需求分析/需求计划表.md`，新增需求记录
4. 状态标记为 🆕 新建

**输出产物**：
- `docs/02-需求分析/需求池/REQ-xxx-<标题>.md`
- 更新的 `docs/02-需求分析/需求计划表.md`

---

### Phase 2: 需求讨论

**触发条件**：用户确认开始讨论某个需求

**执行动作**：
1. 更新需求状态为 💬 讨论中
2. 与用户讨论以下内容：
   - 需求背景和价值
   - 功能范围和边界
   - 用户场景和用例
   - 技术可行性
   - 优先级评估
3. 将讨论结论追加到需求文档的"讨论记录"章节（含时间戳）
4. 确认后更新状态为 ✅ 已确认

**讨论记录格式**：

```markdown
## 讨论记录

### 讨论 1 - YYYY-MM-DD HH:MM

**参与人**：用户, 助手

**讨论内容**：
- 问题 1：xxx
  - 结论：xxx
- 问题 2：xxx
  - 结论：xxx

**待定问题**：
- [ ] 问题 1

**状态**：✅ 已确认 / 🔄 继续讨论
```

**输出产物**：
- 更新的 `REQ-xxx-<标题>.md`（含讨论记录）

---

### Phase 3: 详细设计

**触发条件**：需求已确认，用户要求进行设计

**执行动作**：
1. 在 `docs/03-详细设计/` 创建 `DES-xxx-<标题>.md`
2. 编写设计内容：
   - 功能设计（UI/交互）
   - 数据模型设计
   - API 接口设计
   - 技术方案
   - 安全考虑
   - 测试要点
3. 与用户评审设计
4. 评审通过后更新设计文档

**设计文档格式**：

```markdown
# DES-xxx - <设计标题>

## 基本信息
| 项目 | 内容 |
|------|------|
| 设计编号 | DES-xxx |
| 对应需求 | REQ-xxx |
| 设计人 | 助手 |
| 创建时间 | YYYY-MM-DD HH:MM |
| 评审状态 | 📝 草稿 |

## 一、功能设计
...

## 二、数据模型设计
...

## 三、API 接口设计
...

## 评审记录

### 评审 - YYYY-MM-DD HH:MM
**参与人**：用户, 助手
**评审意见**：...
**评审结论**：✅ 通过
```

**输出产物**：
- `docs/03-详细设计/DES-xxx-<标题>.md`
- 可能更新的 `docs/03-详细设计/API接口文档.md`
- 可能更新的 `docs/03-详细设计/系统架构.md`

---

### Phase 4: 任务分解

**触发条件**：设计完成，准备开发

**执行动作**：
1. 将设计分解为具体开发任务
2. 更新 `docs/04-开发/开发计划表.md`：
   - 添加任务记录
   - 关联需求编号
   - 预估工时
   - 确定优先级
3. 与用户确认开发顺序

**输出产物**：
- 更新的 `docs/04-开发/开发计划表.md`

---

### Phase 5: 开发实施

**触发条件**：用户确认开始开发

**执行动作**：
1. 更新任务状态为 🚧 开发中，记录开始时间
2. 按任务逐项开发：
   - 后端 API 开发
   - 前端页面开发
   - 数据库变更
   - 配置更新
3. 完成后更新任务状态为 🧪 测试中，记录完成时间

**开发日志格式**：

```markdown
### [HH:MM] 开始任务 T001

执行操作：
- 创建 xxx API
- 修改 xxx.vue

### [HH:MM] 遇到问题

问题描述：xxx
解决方案：xxx

### [HH:MM] 完成任务 T001
```

**输出产物**：
- 代码变更（backend/, frontend/）
- 可能新增的技术文档

---

### Phase 6: 测试验证

**触发条件**：开发任务完成

**执行动作**：
1. 执行功能测试
2. 执行 API 测试
3. 记录测试结果（含时间戳）
4. 修复发现的问题
5. 全部通过后更新任务状态为 ✅ 已完成

**测试日志格式**：

```markdown
# 测试日志 - YYYY-MM-DD

## [HH:MM] API 测试

| API | 结果 | 备注 |
|-----|------|------|
| GET /api/xxx | ✅ 通过 | |
| POST /api/xxx | ❌ 失败 | 错误: xxx |

## [HH:MM] 功能测试

| 功能 | 结果 | 备注 |
|------|------|------|
| 用户登录 | ✅ 通过 | |
| 数据保存 | ✅ 通过 | |

## 问题记录

### [HH:MM] 问题 1
- 描述：xxx
- 解决：xxx
- 验证：✅ 已修复
```

**输出产物**：
- 更新的 `docs/07-项目管理/测试日志/YYYY-MM-DD.md`
- 更新的 `docs/05-测试/Bug管理记录.md`（如有 Bug）

---

## 状态流转

### 需求状态

```
🆕 新建 → 💬 讨论中 → ✅ 已确认 → 📋 已规划 → (进入开发)
                ↓
            ❌ 已取消
```

### 任务状态

```
📋 待开发 → 🚧 开发中 → 🧪 测试中 → ✅ 已完成
                            ↓
                        📋 待修复 → 🚧 开发中
```

## 时间记录规范

### 格式要求

1. **日期格式**：`YYYY-MM-DD`
2. **时间格式**：`HH:MM`（24小时制）
3. **时间戳格式**：`YYYY-MM-DD HH:MM`

### 必须记录时间的场景

| 场景 | 时间格式 | 示例 |
|------|----------|------|
| 需求提出 | YYYY-MM-DD | 2026-04-01 |
| 需求讨论 | YYYY-MM-DD HH:MM | 2026-04-01 09:30 |
| 设计创建/评审 | YYYY-MM-DD HH:MM | 2026-04-01 14:00 |
| 任务开始/完成 | HH:MM | 09:30, 17:00 |
| 测试执行 | HH:MM | 15:30 |
| 对话记录 | HH:MM | 10:15 |
| 里程碑达成 | HH:MM | 17:30 |
| Bug 发现/修复 | HH:MM | 11:00 |

## 沟通规范

### 每次对话开始时

1. 检查当前是否有进行中的任务
2. 确认用户意图：新需求 / 继续开发 / 其他
3. 记录开始时间

### 每次对话结束时

1. 确认下一步行动
2. 更新相关文档状态
3. 记录结束时间

### 里程碑完成时

完整执行 Phase 7 里程碑记录流程，所有记录必须包含时间戳

---

## 注意事项

1. **文档优先**：所有工作都要有文档记录
2. **及时更新**：状态变更及时更新计划表
3. **完整追溯**：对话记录要完整保留关键决策，必须包含时间戳
4. **版本管理**：代码变更通过 git 管理，文档变更同样重要
5. **时间记录**：所有操作都必须记录时间，便于追溯和分析
