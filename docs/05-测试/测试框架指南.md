# 测试框架指南

## 概述

OpenClaw Control UI 后端使用 pytest + PostgreSQL 测试数据库进行测试。

---

## 测试工作流程

### ⚠️ 重要原则：发现问题先报告，确认后再修复！

```
1. 运行测试 → 发现失败
       ↓
2. 分析错误原因
       ↓
3. 报告问题（不直接修改代码）
       ↓
4. 等待用户确认
       ↓
5. 确认后再修复
```

### 问题分类和处理方式

| 问题类型 | 处理方式 | 说明 |
|----------|----------|------|
| 测试框架问题 | 报告 → 确认 → 修复 | Fixture、Session 管理等 |
| 测试代码问题 | 报告 → 确认 → 修复 | 断言、数据准备等 |
| 业务代码 Bug | 报告 → 确认 → 修复 | API 错误、逻辑问题等 |
| 功能缺失 | 报告 → 标记跳过 | 未实现的函数 |
| 外部依赖 | 报告 → 调整断言 | Gateway、第三方服务等 |

### 报告格式

发现问题时，使用以下格式报告：

```
## 发现问题

**测试**: `test_xxx`
**错误**: 状态码/异常信息
**原因**: 分析的错误原因
**建议修复方案**: 具体修复建议

是否需要修复？
```

## Fixture 使用

### 核心 Fixture

| Fixture | 作用 | 范围 |
|---------|------|------|
| `db_engine` | 数据库引擎，创建表 | session |
| `db_session` | 数据库 Session，事务回滚 | function |
| `client` | Flask 测试客户端 | function |
| `admin_token` | 管理员 Token | function |
| `admin_user` | 管理员用户 | function |
| `default_roles` | 默认角色 | function |
| `test_department` | 测试部门 | function |

### 使用示例

```python
import pytest

class TestMyFeature:
    """我的功能测试"""

    @pytest.mark.api
    def test_get_list(self, client, admin_token):
        """测试获取列表"""
        response = client.get('/api/my-feature',
            headers={'Authorization': f'Bearer {admin_token}'}
        )
        assert response.status_code == 200

    @pytest.mark.api
    def test_create_item(self, client, admin_token, db_session):
        """测试创建项目"""
        response = client.post('/api/my-feature',
            headers={'Authorization': f'Bearer {admin_token}'},
            json={'name': 'test_item'}
        )
        assert response.status_code in [200, 201]
```

## 测试编写流程

### 新功能开发流程

```
1. 分析需求 → 列出测试场景
2. 创建测试文件 tests/test_<模块名>.py
3. 编写测试用例（覆盖：正常、边界、异常）
4. 实现功能代码
5. 运行测试验证
6. 检查覆盖率，补充缺失测试
```

### 测试场景清单

每个功能模块应包含以下测试类型：

| 测试类型 | 场景示例 |
|----------|----------|
| **列表查询** | 获取成功、分页、搜索、排序 |
| **创建** | 创建成功、重复数据、缺少字段、无效字段 |
| **详情** | 获取成功、不存在记录 |
| **更新** | 更新成功、不存在记录、无效字段 |
| **删除** | 删除成功、不存在记录、关联约束 |
| **权限** | 无 Token、无效 Token、权限不足 |
| **边界** | 空值、超长、特殊字符、极限值 |

## 常见问题

### 1. 用户名/ID 冲突

多个测试同时运行可能导致唯一字段冲突。

**解决方法：** 使用时间戳生成唯一值

```python
import time
unique_name = f'user_{int(time.time() * 1000)}'
```

### 2. PostgreSQL 序列问题

使用固定 ID 创建数据后，后续自动生成的 ID 可能冲突。

**解决方法：** 让数据库自动生成 ID，不要设置固定 ID

```python
# 错误：设置固定 ID
user = User(id=1, username='test')  # 可能导致序列不同步

# 正确：让数据库自动生成
user = User(username='test')
db_session.add(user)
db_session.commit()
db_session.refresh(user)  # 获取自动生成的 ID
```

### 3. 测试 Session 隔离

测试之间 Session 可能互相干扰，导致数据不一致。

**解决方法：** 使用 `set_test_session()` 和 `clear_test_session()`

```python
# 在 conftest.py 中
@pytest.fixture
def app(db_session):
    from database import db as global_db
    global_db.set_test_session(db_session)  # 设置测试模式
    yield flask_app
    global_db.clear_test_session()  # 清理
```

测试模式下，DatabaseCompat 不会在操作后关闭 Session，保持测试事务的完整性。

### 4. API 返回状态码不一致

某些 API 返回 200 而非 201，或对不存在记录返回 200 而非 404。

**解决方法：** 使用灵活断言

```python
# 创建操作
assert response.status_code in [200, 201]

# 不存在记录
assert response.status_code in [200, 404, 400]

# API 不存在
assert response.status_code in [200, 404, 500]
```

### 3. 外键约束

删除有关联数据的记录时会失败。

**解决方法：**

```python
# 方式一：创建无关联的测试数据
dept = Department(name='独立部门')
db_session.add(dept)
db_session.commit()

# 方式二：断言允许失败
assert response.status_code in [200, 400, 404]
```

### 4. 速率限制

登录 API 可能触发速率限制（429）。

**解决方法：**

```python
if login_response.status_code == 429:
    pytest.skip("Rate limiting triggered")
```
