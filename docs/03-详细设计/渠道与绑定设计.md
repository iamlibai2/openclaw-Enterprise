# 渠道与绑定设计文档

## 一、背景

OpenClaw 支持多渠道接入（飞书、钉钉等），通过 **Channels（渠道）** 和 **Bindings（绑定）** 配置消息路由：

- **Channels**：定义外部通信渠道及其账号配置
- **Bindings**：定义消息到 Agent 的路由规则

当前 Admin UI 的绑定配置功能较为简陋，需要完善以支持完整的 CRUD 操作和更丰富的匹配规则。

## 二、现状分析

### 2.1 配置结构

**Channels（渠道配置）**：

```json
{
  "channels": {
    "dingtalk-connector": {
      "enabled": true,
      "accounts": {
        "pili": {
          "clientId": "ding2tqzqxx3d9wem8k6",
          "clientSecret": "...",
          "gatewayToken": "..."
        }
      },
      "defaultAccount": "pili",
      "sharedMemoryAcrossConversations": true
    },
    "feishu": {
      "enabled": true,
      "threadSession": true,
      "accounts": {
        "default": { "appId": "...", "appSecret": "..." },
        "xiaomei": { "appId": "...", "appSecret": "..." },
        "aqiang": { "appId": "...", "appSecret": "..." },
        "dazhuang": { "appId": "...", "appSecret": "..." }
      }
    }
  }
}
```

**Bindings（绑定规则）**：

```json
{
  "bindings": [
    { "agentId": "main", "match": { "channel": "dingtalk-connector", "accountId": "pili" } },
    { "agentId": "xiaomei", "match": { "channel": "feishu", "accountId": "xiaomei" } },
    { "agentId": "aqiang", "match": { "channel": "feishu", "accountId": "aqiang" } },
    { "agentId": "dazhuang", "match": { "channel": "feishu", "accountId": "dazhuang" } }
  ]
}
```

### 2.2 当前问题

| 问题 | 现状 |
|------|------|
| Bindings 管理 | 只能创建，不能编辑/删除/排序 |
| 字段不匹配 | 前端使用 `channelType/channelId/chatId`，实际应为 `channel/accountId/peer` |
| 规则不完整 | 不支持 `peer.kind`（direct/group）和 `peer.id` |
| Channels 管理 | 没有专门页面，只能通过 JSON 编辑器修改 |
| 账号管理 | 无法在 UI 中添加/编辑/删除 channel accounts |
| 可视化差 | 无法直观看到 Agent-Channel-Account 的绑定关系 |

## 三、核心概念

### 3.1 三层架构

```
┌─────────────────────────────────────────────────────────┐
│                     员工 (Employee)                      │
│    (Agent 的"人格化"，有名称、部门、头像等)               │
└─────────────────────┬───────────────────────────────────┘
                      │ 绑定 (employees.agent_id)
                      ▼
┌─────────────────────────────────────────────────────────┐
│                     Agent                                │
│    (工作节点，有 workspace、model、skills 等)            │
└─────────────────────┬───────────────────────────────────┘
                      │ Bindings 路由
                      ▼
┌─────────────────────────────────────────────────────────┐
│              Channel + Account                           │
│    (外部通信渠道 + 账号配置)                             │
│    feishu/xiaomei  feishu/aqiang  dingtalk/pili         │
└─────────────────────────────────────────────────────────┘
```

### 3.2 Binding 匹配规则

根据 OpenClaw Gateway 规范，binding 的完整 match 结构：

```typescript
interface Binding {
  agentId: string
  match: {
    channel?: string              // 频道名：feishu, dingtalk-connector
    accountId?: string            // 账号 ID：对应 channel.accounts 中的 key
    peer?: {
      kind?: 'direct' | 'group'   // 会话类型
      id?: string                 // 群ID 或 用户ID，"*" 表示通配
    }
  }
}
```

**匹配逻辑**：

1. 按 `bindings` 数组顺序遍历
2. 所有指定的维度必须同时满足（AND 关系）
3. 第一条命中的规则生效
4. 未命中任何规则则使用 `defaultAgent`

**匹配优先级**：

- 精确规则放前面（指定 `peer.id`）
- 通配规则放后面（`peer.id: "*"`）

### 3.3 会话隔离配置

Channel 级别的会话隔离配置：

| 配置 | 说明 |
|------|------|
| `sharedMemoryAcrossConversations` | 所有会话共享记忆（单聊+群聊） |
| `separateSessionByConversation` | 按会话类型区分 session |
| `groupSessionScope` | 群聊 session 粒度：`group` 或 `group_sender` |
| `threadSession` | 飞书专属，按消息串隔离 |

## 五、API 设计

### 5.1 Bindings API

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/bindings` | 获取绑定列表 |
| POST | `/api/bindings` | 创建绑定 |
| PUT | `/api/bindings/:index` | 更新绑定（按索引） |
| DELETE | `/api/bindings/:index` | 删除绑定 |
| PUT | `/api/bindings/order` | 调整绑定顺序 |

### 5.2 Channels API

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/channels` | 获取所有渠道 |
| GET | `/api/channels/:channel` | 获取单个渠道配置 |
| PUT | `/api/channels/:channel` | 更新渠道配置 |
| POST | `/api/channels/:channel/accounts` | 添加账号 |
| PUT | `/api/channels/:channel/accounts/:accountId` | 更新账号 |
| DELETE | `/api/channels/:channel/accounts/:accountId` | 删除账号 |
| POST | `/api/channels/:channel/test` | 测试渠道连接 |

### 5.3 WebSocket 方法映射

所有配置读写通过 Gateway WebSocket：

| 操作 | WebSocket 方法 |
|------|---------------|
| 读取配置 | `config.get` |
| 更新配置 | `config.patch` |

## 六、数据模型

### 6.1 Binding 类型定义

```typescript
interface Binding {
  agentId: string
  match: {
    channel?: string
    accountId?: string
    peer?: {
      kind?: 'direct' | 'group'
      id?: string
    }
  }
}
```

### 6.2 Channel 类型定义

```typescript
interface Channel {
  enabled: boolean
  accounts: Record<string, ChannelAccount>
  defaultAccount?: string
  // 会话配置
  threadSession?: boolean
  sharedMemoryAcrossConversations?: boolean
  separateSessionByConversation?: boolean
  groupSessionScope?: 'group' | 'group_sender'
  // 权限配置
  dmPolicy?: 'open' | 'closed'
  groupPolicy?: 'open' | 'closed'
  allowFrom?: string[]
  groupAllowFrom?: string[]
  requireMention?: boolean
}

interface ChannelAccount {
  enabled?: boolean
  appId?: string        // 飞书
  appSecret?: string    // 飞书
  clientId?: string     // 钉钉
  clientSecret?: string // 钉钉
  gatewayToken?: string // 钉钉
  groupAllowFrom?: string[]
  groupPolicy?: string
}
```

## 八、设计决策

以下决策已确认：

### 8.1 绑定粒度

**决策**：当前阶段只支持到 `channel + accountId` 级别，不支持 `peer.id` 精细绑定。

**理由**：
- 简化 UI 和用户操作
- 大多数场景按账号级别路由已足够
- 未来有需求时可扩展

### 8.2 敏感信息处理

**决策**：脱敏显示 + 点击显示

**实现方式**：
- 默认显示 `sk-****abcd`（首尾各显示部分字符）
- 点击"显示"按钮后显示完整内容
- 前端不做二次确认（敏感操作由权限系统控制）

### 8.3 默认 Agent 配置

**决策**：在绑定配置页面明确显示和配置 `defaultAgent`

**UI 位置**：绑定列表底部，显示当前默认 Agent，允许下拉选择切换

### 8.4 频道扩展性

**决策**：设计需考虑未来扩展，支持更多渠道类型

**实现方式**：
- 渠道类型不硬编码，从 `config.get` 返回的 channels 动态读取
- 每个渠道类型可有不同的账号字段配置（飞书: appId/appSecret，钉钉: clientId/clientSecret）
- 渠道卡片设计通用布局，特殊配置通过"高级设置"折叠展示

**未来可能支持的渠道**：
- 企业微信
- Slack
- Discord
- 微信公众号/小程序

### 8.5 绑定页面位置

**决策**：绑定配置作为独立页面，菜单位置：`OpenClaw 管理` → `绑定配置`

**理由**：
- 集中管理所有路由规则
- 便于查看整体路由拓扑
- 与 Agent 管理、配置编辑并列，符合"配置类"功能定位
