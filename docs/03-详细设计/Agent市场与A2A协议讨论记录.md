## 二、A2A 协议调研

### 2.1 什么是 A2A 协议

**Agent-to-Agent Protocol**，由 Google 于 2025年4月发布，2025年6月移交 Linux 基金会管理。

**核心定位**：
> 让不同框架、不同供应商构建的 AI Agent 之间能够安全、标准化地通信与协作。

**核心组件**：

| 组件 | 说明 |
|------|------|
| **Agent Card** | JSON 格式的"数字身份证"，包含能力、端点、认证要求 |
| **Task Object** | 标准化的任务对象，有完整生命周期 |
| **SSE** | Server-Sent Events，实时状态推送 |
| **JSON-RPC** | 通信协议 |

**Agent Card 端点**：
```
/.well-known/agent.json
```

### 2.2 A2A vs MCP

| 协议 | 解决的问题 | 类比 |
|------|-----------|------|
| **A2A** | Agent 之间的协作 | "我指挥你干活" |
| **MCP** | Agent 与工具/数据源的连接 | "我亲自干活，工具辅助" |

**结论**：两者互补，不是竞争关系。A2A 是"协作层"，MCP 是"工具层"。

### 2.3 技术实现参考

- **阿里 AgentScope**：已支持 A2A 协议，集成 Nacos 作为 Registry
- **Google ADK**：官方开发工具包
- **开源仓库**：https://github.com/a2aproject/A2A

## 三、RaaS 商业模式调研

### 3.1 什么是 RaaS

**Result-as-a-Service（结果即服务）**：
- 交付的是"结果"，不是"工具"
- 用户按效果付费，不是按订阅付费
- 用户无需学习配置，只需"雇佣" Agent

### 3.2 成功案例

#### 案例一：光年触达 - iSales（AI 销售 Agent）

| 指标 | 数据 |
|------|------|
| 上线时间 | 2025年4月 |
| 现金流转正 | 1个月 |
| 月营收 | 200万元（2026年1月） |
| 预计全年 | 5000万元 |
| 获客效率提升 | 10倍 |
| 营销转化率 | 8% |

**核心逻辑**：
> "我们交付的不是工具，是结果。客户无需额外招人、学系统操作，iSales 自动完成客户开发、邮件触达、意向筛选等工作，企业只需跟进最终筛选出的有效线索，并按实际效果付费。"

#### 案例二：百融智能 - 10,000+ 企业级 Agent

| 指标 | 数据 |
|------|------|
| 年收入 | 29.2亿元 |
| 毛利率 | 72% |
| 核心客户留存率 | 98%（历史最高） |
| Agent 数量 | 10,000+ |

**演进路径**：
```
MaaS（模型即服务）→ BaaS（业务即服务）→ RaaS（结果即服务）
```

**案例场景**：
- 鱼泡直聘（蓝领招聘）：VoiceAgent 作为"硅基招聘官"
- 阿里国际站 Accio Work：40% 用户是一人企业

### 3.3 RaaS vs SaaS

| 维度 | SaaS | RaaS |
|------|------|------|
| 交付物 | 工具/平台 | 结果 |
| 付费方式 | 订阅制 | 按效果付费 |
| 用户门槛 | 需要学习配置 | 即插即用 |
| 价值衡量 | 功能多少 | 业务结果 |

## 五、架构讨论：双系统设计

### 5.1 用户的核心观点

> "我们的项目主要供企业内部使用。Agent 市场可以是另一个项目：一个是企业内部系统，一个是云端市场，两者结合。"

### 5.2 架构图

```
┌─────────────────────────────────────────────────────────────┐
│                        云端市场                               │
│  (独立项目 - 开放平台)                                        │
│                                                              │
│  • Agent 模板商店                                            │
│  • 社区贡献/评分                                              │
│  • 企业认证模板                                               │
│  • A2A 协议注册中心                                           │
└─────────────────────────────────────────────────────────────┘
                              ▲
                              │ 下载/上传
                              │
┌─────────────────────────────────────────────────────────────┐
│                    企业内部系统                               │
│  (当前项目 - Control UI)                                     │
│                                                              │
│  • Agent 管理/运行                                           │
│  • 企业数据/知识库                                            │
│  • 权限/审计                                                  │
│  • 内部 Agent 协作                                            │
└─────────────────────────────────────────────────────────────┘
```

### 5.3 两系统对比

| 维度 | 企业内部系统 (Control UI) | 云端市场 |
|------|---------------------------|----------|
| **用户** | 企业员工 | 所有人/企业 |
| **数据** | 企业私有，不出域 | 公开分享 |
| **主要功能** | 运行/管理 Agent | 发现/分享 Agent |
| **商业模式** | 企业部署服务 | 平台服务/模板交易 |
| **安全要求** | 高（内网/私有化） | 中（公开平台） |

### 5.4 结合方式

1. **从市场下载**
   - 企业管理员浏览云端市场
   - 选择合适的 Agent 模板
   - 一键导入到企业内部系统
   - 模板不包含敏感数据

2. **向市场贡献**
   - 企业调试好一个 Agent
   - 脱敏后上传到市场
   - 可选择公开或仅企业可见

3. **A2A 协议**
   - 云端市场作为 A2A Agent 注册中心
   - 企业 Agent 可选择是否暴露到公网
   - 企业间 Agent 协作通过市场撮合

## 六、RaaS 内部结算机制（重点方向）

### 6.1 核心洞察

> **用户反馈**："内部结算这个提议真的太棒了！也会让内部的员工更有积极性来使用 Agent。"

这是 RaaS 模式在企业内部应用的关键价值点。

### 6.2 传统模式的问题

```
┌─────────────────────────────────────────┐
│  传统模式                                │
│                                         │
│  IT部门（成本中心）                      │
│       │                                 │
│       ↓ 开发、维护                       │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  │
│  │ 销售部  │  │ 财务部  │  │ 人事部  │  │
│  └─────────┘  └─────────┘  └─────────┘  │
│       │            │           │        │
│       └────────────┴───────────┘        │
│                    │                    │
│              "IT做的什么？"              │
│              "不知道，反正预算批了"      │
└─────────────────────────────────────────┘
```

**问题**：
- IT 是"成本中心"，只花钱不赚钱
- 业务部门不知道 IT 的价值
- Agent 开发缺乏反馈，不知道哪个好用
- 员工没有动力去尝试新工具

### 6.3 内部结算模式

```
┌─────────────────────────────────────────┐
│  内部结算模式                            │
│                                         │
│  IT部门（服务中心）                      │
│       │                                 │
│       │ 提供 Agent 服务                  │
│       ↓                                 │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  │
│  │ 销售部  │  │ 财务部  │  │ 人事部  │  │
│  │ 预算￥  │  │ 预算￥  │  │ 预算￥  │  │
│  └─────────┘  └─────────┘  └─────────┘  │
│       │            │           │        │
│       └────────────┴───────────┘        │
│                    │                    │
│              按效果"付费"给 IT           │
└─────────────────────────────────────────┘
```

**好处**：

| 角度 | 好处 |
|------|------|
| **业务部门** | 有预算控制权，会选择真正好用的 Agent |
| **IT 部门** | 有"收入"，知道哪些 Agent 价值高 |
| **员工** | 用好 Agent = 省预算 = 更高绩效 |
| **企业** | 资源配置透明，知道钱花在哪 |

### 6.4 员工积极性的来源

1. **预算所有权** - "这笔钱是我的，我要花在刀刃上"
2. **选择权** - "我可以选择雇佣哪个 Agent"
3. **效果可见** - "我花了钱，能看到实实在在的效果"
4. **绩效挂钩** - "用 Agent 省下的预算，可以转化为我的绩效"

### 6.5 类似管理模式参考：阿米巴经营

稻盛和夫的**阿米巴经营**理念：

> 每个部门都是独立的核算单元，都要算账。部门之间的服务要"交易"，让价值流动起来。

**应用到 Agent**：

```
销售部预算：10万/月
├── 雇佣"销售助手Agent"：2万/月
│   └── 完成线索筛选：5000条 × 4元/条
├── 雇佣"邮件Agent"：1万/月
│   └── 发送邮件：10000封 × 1元/封
└── 剩余预算：7万（用于其他用途）

IT 部门收入：3万/月
└── 有收入就能招聘更多人、优化 Agent
```

### 6.6 待细化问题

| 问题 | 可能的方案 |
|------|-----------|
| **计费方式** | 按任务数？按成功率？按节省时间？按业务结果？ |
| **预算来源** | 业务部门自己申请？公司统一分配？ |
| **结算周期** | 按月？按季度？实时？ |
| **IT 定价权** | IT 能自己定价吗？还是公司统一定价？ |
| **虚拟货币** | 用虚拟积分还是真实金额？ |
| **绩效考核** | 如何与员工绩效挂钩？ |

### 6.7 设计思路

```
┌──────────────────────────────────────────────────────┐
│                   内部结算系统                        │
│                                                      │
│  ┌────────────┐    ┌────────────┐    ┌────────────┐  │
│  │ Agent 计费 │    │ 预算管理   │    │ 结算报表   │  │
│  │ 模块       │    │ 模块       │    │ 模块       │  │
│  └────────────┘    └────────────┘    └────────────┘  │
│         │                │                 │         │
│         └────────────────┴─────────────────┘         │
│                          │                           │
│                   ┌──────↓──────┐                    │
│                   │  数据统计   │                    │
│                   │  • 调用量   │                    │
│                   │  • 成功率   │                    │
│                   │  • 耗时     │                    │
│                   └─────────────┘                    │
└──────────────────────────────────────────────────────┘
```

**核心功能**：
1. 每个 Agent 有明确的"服务价格"
2. 每个部门有"Agent 预算"
3. 每次调用自动计费
4. 月底生成结算报表

## 七、关键洞察

### 7.1 为什么 RaaS 成立

> "传统 SaaS 交付工具，企业购买后需自行配置人员使用。现实是软件买回去后缺乏专人持续运营，难以产生实际价值。Agent 模式交付的是**结果**。"

### 7.2 一人企业（Solopreneur）是重要用户群

- 阿里国际站 Accio Work：40% 用户是一人企业
- 他们需要的是"即插即用"的能力，而不是需要学习的工具

### 7.3 A2A 协议作为 Agent 市场的技术基础

Agent Card 可以成为**标准化的 Agent 模板格式**：
- 用户从市场"下载"一个 Agent Card
- 系统自动解析、配置、部署
- Agent 市场不只是"配置分享"，而是**标准化、可互操作的能力交换**
