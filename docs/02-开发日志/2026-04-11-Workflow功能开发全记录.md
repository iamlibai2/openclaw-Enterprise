# Workflow 功能开发全记录 - 2026-04-10 ~ 2026-04-11

## 一、功能概述

**目标**：实现 Skill 工作流编排功能，用户通过自然语言创建和执行工作流。

**核心原则**：
- 自然语言操作，可视化只呈现
- 人不在流程中，人在流程外（只做事后评估）

## 二、设计讨论

### 2.1 架构设计的讨论（2026-04-10）

**原始设计问题**：
- Backend 调用 Skill，但 Skill 在 Gateway 上运行
- 形成"Backend → Gateway → Skill"的循环调用

**讨论过程**：

> 用户：backend 为什么要调用gateway 之前是怎么设计的

> 回顾设计文档后发现：
> - 编排 Agent（Prometheus）在 Gateway 上运行
> - 工作流引擎"调用 Skill"，但没有明确在哪里调用

**确定的新架构**：

```
Backend = 编排大脑（只做拓扑排序、变量解析、返回指令）
Prometheus = 执行手脚（接收指令 → 执行 Skill → 提交结果 → 继续循环）
```

**核心改动**：
- Backend 不执行 Skill，只返回"下一步执行什么"
- Prometheus 循环调用编排 API，自己执行 Skill

### 2.2 DAG 状态更新的讨论（2026-04-11）

> 用户：那dag 的状态更新呢

**问题**：Prometheus 执行节点时，前端不知道实时状态。

**讨论方案**：
- 方案 A：通过对话输出节点状态标记（最简单）
- 方案 B：Gateway 推送 WebSocket 事件（需要额外开发）

**选择方案 A**：Prometheus 输出 `[NODE_START]`, `[NODE_COMPLETE]` 等标记，前端解析。

### 2.3 执行参数弹窗的讨论

**问题**：API 返回的 `userInputSchema` 没有 `required` 字段。

**解决**：没有 `default` 值的字段就是必需参数。

**问题**：切换页面时 streaming 停止

**解决**：
- Gateway 连接不随组件销毁停止
- 切换回来重新绑定事件监听器
- sessionKey 保持不变

```python
def execute_workflow(name: str, user_input: dict) -> dict:
    # 开始编排
    result = orchestrate_start(name, user_input)
    current_instruction = result["data"]

    while current_instruction.get("status") == "running":
        node_id = current_instruction["node_id"]
        skill_id = current_instruction["skill"]
        input_data = current_instruction["input"]

        # 输出节点状态标记
        print(f"[NODE_START: {node_id}]")

        # 执行 Skill
        skill_result = execute_skill(skill_id, input_data)

        print(f"[NODE_COMPLETE: {node_id}]")

        # 提交结果，获取下一个指令
        result = orchestrate_submit_result(execution_id, node_id, output)
        current_instruction = result["data"]

    print(f"[EXECUTION_COMPLETE: {name}]")
```

### 7.3 前端解析节点状态

```typescript
function parseWorkflowResponse(text: string) {
  const nodeStartMatches = text.matchAll(/\[NODE_START:\s+(\S+)\]/g)
  for (const match of nodeStartMatches) {
    const nodeId = convertNodeId(match[1])  // node_1 → node-0
    workflowStore.updateNodeStatus(nodeId, 'running')
  }
}

function convertNodeId(backendNodeId: string): string {
  const match = backendNodeId.match(/node_(\d+)/)
  if (match) {
    return `node-${parseInt(match[1]) - 1}`
  }
  return backendNodeId
}
```

## 十、核心设计理解：Agent 灵活执行模型

### 10.1 背景

工作流节点执行不应该依赖固定的数据格式标准，因为：

1. **Skill 是第三方的**：输出格式不可控
2. **数据格式多样**：JSON、Markdown、Word、PDF、文件路径、URL...
3. **Agent 的核心能力**：灵活理解、做格式转换、智能处理

### 10.2 执行权交接模型

```
┌─────────────────────────────────────────────────────────────┐
│  执行流程                                                    │
│                                                             │
│  用户 → Prometheus (持有执行权)                              │
│         ↓                                                    │
│  skill 节点 → Prometheus 直接执行 Skill                      │
│         ↓                                                    │
│  skill 节点 → Prometheus 直接执行 Skill                      │
│         ↓                                                    │
│  agent 节点 → Prometheus 把执行权交接给 agent_zhangsan        │
│         ↓                                                    │
│  agent_zhangsan (持有执行权)                                 │
│         ↓                                                    │
│  skill 节点 → agent_zhangsan 直接执行 Skill                  │
│         ↓                                                    │
│  agent 节点 → agent_zhangsan 把执行权交接给下一个 Agent       │
│         ↓                                                    │
│  ... 继续执行 ...                                            │
└─────────────────────────────────────────────────────────────┘
```

**核心规则**：
- `type: "skill"` → 当前持有执行权的 Agent 直接执行 Skill
- `type: "agent"` → 执行权交接给指定 Agent，该 Agent 继续执行后续节点

### 10.3 Prometheus 的思考过程

真正"灵活理解 Skill 定义、做格式转换"是 Prometheus Agent 在对话中思考处理：

```
用户: 执行工作流 search-write-publish，主题是 AI 最新动态

Prometheus:
  1. 调用 Backend /orchestrate/start
     → 获取第一个节点: {node_1, skill: baidu-search}

  2. 思考:
     - baidu-search 需要什么？看 Skill 定义...
     - 需要 query 参数，用户给了主题 "AI 最新动态"
     - 调用 baidu-search Skill

  3. 获取输出: [{title, url, content}, ...]
     → 思考: 下一个节点 article-generator 需要什么？
     - 看 article-generator 的 Skill 定义...
     - 需要 markdown 输入，但上一步是 JSON 数组
     - 做格式转换: 把 JSON 数组拼成 markdown 文本
     - 调用 article-generator Skill

  4. 获取输出: {"file_path": "/tmp/article.md"}
     → 思考: 下一个节点是什么？
     - Backend 返回 agent 节点
     - 执行权交接给 agent_zhangsan
     - 告知 zhangsan: "请继续执行工作流，从节点3开始，文件在 /tmp/article.md"
```

### 10.4 Backend API 设计

| API | 说明 | 返回 |
|------|------|------|
| `/orchestrate/start` | 开始执行 | 第一个节点指令 |
| `/orchestrate/result` | 提交节点结果 | 下一个节点指令或完成状态 |
| `/orchestrate/error` | 提交节点错误 | 失败状态 |
| `/orchestrate/status/<id>` | 获取执行状态 | 当前进度 |
| `/orchestrate/continue` | Agent 交接后继续 | 从指定节点继续执行 |

### 10.5 Agent 交接指令格式

```json
{
  "status": "running",
  "execution_id": "exec_xxx",
  "node_id": "node_3",
  "type": "agent",
  "action": "handover",
  "to_agent": "agent_zhangsan",
  "workflow_name": "search-write-publish",
  "from_index": 3,
  "context": {
    "previous_outputs": {...},
    "user_input": {...}
  }
}
```

### 10.6 与传统工作流引擎的区别

| 传统引擎 | Agent 编排引擎 |

## 十三、Skill 执行逻辑修正（2026-04-11）

### 13.1 问题发现

**错误理解**：
- 认为 Skill 是 Gateway 上的能力，需要通过 Gateway 协议调用
- execute.py 写死了查找 .py 文件，不支持其他语言

**正确理解（查阅官方文档后）**：
- Skill 是 Agent 工作区的一部分（`workspace/skills/` 目录）
- SKILL.md 是给 LLM 的指令文档，被注入到系统提示词
- scripts 目录是可选辅助脚本，Agent 自己决定如何调用
- Gateway 不执行 Skill，只负责消息传递

### 13.2 修正内容

**删除 execute_skill 工具**：
- 不需要固定的 Skill 执行工具
- Prometheus 自己决定如何执行（读取 SKILL.md 后理解）

**保留 read_skill_definition 工具**：
- 用于读取 SKILL.md 内容
- Prometheus 理解后自己选择执行方式

### 13.3 新的执行流程

```
Backend 返回指令 → Prometheus 收到 "执行 baidu-search"
                  ↓
Prometheus 用 read_skill_definition 读取 SKILL.md
                  ↓
Prometheus 理解调用方式：python3 scripts/search.py '<JSON>'
                  ↓
Prometheus 自己用 bash 工具执行
                  ↓
Prometheus 提交结果到 Backend
```

## 十四、Agent 交接机制确认（2026-04-11）

### 14.1 查阅官方文档

查阅 OpenClaw 官方文档（`/usr/lib/node_modules/openclaw/docs`）后，确认两种 Agent 通信机制：

| 方式 | 说明 | 适用场景 |
|------|------|----------|
| `sessions_spawn` | 生成子智能体，后台运行，完成后通告 | 并行任务、不需要等待结果 |
| `sessions_send` | 向另一个 Agent 发送消息，等待回复 | 同步交接、需要立即返回结果 |

### 14.2 选择 sessions_send

**理由**：工作流 Agent 交接需要同步等待结果

```
Prometheus → sessions_send → 小美执行 → 返回结果 → Prometheus 继续编排
```

### 14.3 sessions_send 参数

```python
sessions_send(
  sessionKey: "agent:xiaomei:main",  # 目标 Agent 的主会话
  message: "执行工作流节点 node_3，上下文：{...}",
  timeoutSeconds: 300  # 等待 5 分钟
)
```

返回：
- `{ status: "ok", reply: {...} }` - 执行成功，有回复
- `{ status: "timeout", error }` - 超时
- `{ status: "error", error }` - 执行失败

### 14.4 更新 SOUL.md

添加 Agent 交接详细说明：
- 使用 sessions_send 向指定 Agent 发送消息
- sessionKey 格式：`agent:<agent_id>:main`
- message 包含任务描述、上下文、输入数据
- timeoutSeconds: 300
- 收到回复后用 orchestrate_submit_result 提交结果

### 14.5 sessionKey 格式

| Agent | sessionKey |
|-------|------------|
| 小美 | `agent:xiaomei:main` |
| 阿强 | `agent:aqiang:main` |
| Prometheus | `agent:prometheus:main` |

### 14.6 前提条件

Gateway 配置需要：
```json5
{
  tools: {
    agentToAgent: {
      enabled: true,
      allow: ["prometheus", "xiaomei", "aqiang"]
    }
  }
}
```

### 14.7 限制

目前 sessions_send 只支持同一 Gateway 内的 Agent 通信。跨服务器需要：
- 统一到一个 Gateway（推荐）
- 或通过 HTTP API 实现远程调用

## 十五、配置各 Agent 工作流执行能力（2026-04-11）

### 15.1 设计原则

| Agent | 能力 |
|-------|------|
| Prometheus | 创建 + 执行 + 编排（完整能力） |
| 小美/阿强 | 只执行（接收交接 → 执行节点 → 提交结果） |

**不使用共享 Skill 目录，而是复制到各 Agent 的 skill 目录。**

### 15.2 完成的配置

**1. 复制 workflow-execute Skill**

```
workspace-xiaomei/skills/workflow-execute/
  - SKILL.md（执行者版）
  - scripts/execute.py

workspace-aqiang/skills/workflow-execute/
  - SKILL.md（执行者版）
  - scripts/execute.py
```

**2. SKILL.md 区别**

Prometheus 版：包含创建 + 执行
小美/阿强版：只包含执行（接收交接、执行节点、提交结果）

**3. 更新 openclaw.json**

小美配置：
```json
"skills": ["acp-router", "clawhub", "workflow-execute"],
"tools": {
  "alsoAllow": ["browser", "sessions_list", "sessions_history", "sessions_send", "session_status"]
}
```

阿强配置：
```json
"skills": ["acp-router", "Powerpoint / PPTX", "workflow-execute"]
```

阿强使用 messaging profile，已包含 sessions 工具。

### 15.3 各 Agent 的 workflow-execute 工具

| 工具 | Prometheus | 小美/阿强 |
|------|------------|----------|

### 15.4 交接消息格式

Prometheus 用 sessions_send 发送：
```json
{
  "type": "workflow_handover",
  "execution_id": "exec_xxx",
  "node_id": "node_3",
```

小美/阿强收到后：
1. 执行 Skill
2. 用 orchestrate_submit_result 提交结果
3. 回复执行结果给 Prometheus
