# OpenClaw 员工-Agent-工作流系统设计方案

> 版本：v1.0
> 状态：正式设计
> 创建时间：2026-04-10
> 基于多次讨论修正后的结论

---

## 一、设计背景

### 1.1 现有系统

OpenClaw Control UI 已实现：
- 员工管理（Employee 表）
- Agent 管理（通过 Gateway）
- 员工-Agent 绑定关系（agent_ids 字段）
- Skill 工作流编排功能（P0 完成）

### 1.2 设计目标

将独立的功能整合为完整的工作流系统：
- 员工通过 Agent 发起并执行工作流
- Agent 根据员工偏好自主执行
- 员工验收结果并反馈优化

---

## 二、核心理念

### 2.1 Agent 是员工的能力延伸

```
传统组织：员工 → 下属 → 执行 → 汇报 → 审批
                    ↑ 层级管理、审批链

新设计：员工 ← Agent（能力延伸）
         ↑ 员工受益，Agent赋能
         ↑ 员工只参与两端（需求+验收）
```

### 2.2 人在流程外

**Skill 工作流编排的核心原则**：

```
❌ 旧思路：人在流程里
流程 ──节点1──▶ 节点2 ──人工审批──▶ 节点3 ──人工发布──▶ 完成

✅ 新思路：人在流程外
流程 ──节点1──▶ 节点2 ──节点3──▶ 节点4──▶ 完成
                                      ↓
                             人查看结果、评估、提建议
```

### 2.3 三层架构

```
┌─────────────────────────────────────────────────────────────────┐
│                          员工层                                  │
│                                                                  │
│  • 提出需求（自然语言）                                          │

│  • 反馈优化                                                      │
│                                                                  │
│  角色：受益者，不是管理者                                         │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                          Agent层                                 │
│                                                                  │

│  • 选择/创建工作流                                               │

│  • 汇报结果（根据员工偏好）                                       │
│                                                                  │
│  角色：能力延伸，不是下属                                         │
```

## 三、角色定义

### 3.1 员工角色

| 传统定义 | 新定义 |
|----------|--------|
| 任务分配者 | **需求提出者** |
| 审批者 | **结果验收者** |
| 管理者 | **受益者** |

**员工职责**：
1. 向 Agent 提出工作需求
2. 等待 Agent 执行完成
3. 验收执行结果
4. 提供反馈和优化建议

**员工不参与**：
- 流程中间环节
- 审批决策
- 人工节点

### 3.2 Agent 角色

| 传统定义 | 新定义 |
|----------|--------|
| 执行工具 | **能力延伸** |
| 下属 | **自主执行者** |
| 汇报者 | **赋能者** |

**Agent 职责**：
1. 理解员工需求意图
2. 选择或创建合适的工作流
3. 全自主执行工作流
4. 根据员工偏好生成汇报
5. 学习员工反馈持续改进

## 五、数据库设计

### 5.1 Employee 表结构（最终方案）

采用**混合方案**：核心字段单独存 + 复杂配置存JSON

| 字段 | 类型 | 用途 | 备注 |
|------|------|------|------|
| `agent_ids` | TEXT | 绑定关系 | JSON数组，保持不变 |
| `agent_config` | JSONB | 能力配置 | 新增字段 |

### 5.2 字段分工

```
agent_ids（绑定关系）
──────────────────────────
存储：["agent_001", "agent_002"]
用途：员工绑定了哪些 Agent
查询：经常按 Agent ID 查询员工

agent_config（能力配置）
──────────────────────────
存储：{ autonomy, report_style, learning }
用途：Agent 如何为员工服务
查询：不常单独查询，灵活扩展
```

### 5.3 agent_config 数据结构

```json
{
  "autonomy": "high",
  "report_style": {
    "detail_level": "summary",
    "timing": "on_complete"
  },
  "learning": {
    "remember_feedback": true,
    "auto_improve": true
  }
}
```

**字段说明**：

| 字段 | 类型 | 说明 | 可选值 |
|------|------|------|--------|
| `autonomy` | string | 自主性级别 | high / medium / low |
| `report_style.detail_level` | string | 汇报详细程度 | summary / detail |
| `report_style.timing` | string | 汇报时机 | on_complete / realtime |
| `learning.remember_feedback` | boolean | 记住员工反馈 | true / false |
| `learning.auto_improve` | boolean | 自动学习改进 | true / false |

### 5.4 自主性级别详解

| 级别 | 说明 | 行为 |
|------|------|------|
| **high** | 高度自主 | 不需要确认就执行；完成后汇报；失败自动重试 |
| **medium** | 中等自主 | 需要确认后执行；实时汇报进度；失败询问员工 |
| **low** | 低自主 | 每步确认；每步汇报；失败询问员工 |

## 六、工作流存储设计

### 6.1 存储方式

采用 **Markdown + 内嵌 JSON** 格式，不使用数据库。

**设计原则**：
- JSON 部分：核心结构数据（节点、边、变量绑定）
- Markdown 部分：人类可读描述、说明、文档

### 6.2 文件格式

```markdown
# 公众号文章发布

创建时间: 2026-04-10
状态: ready

## 目标

搜索 AI 相关资料，生成正式风格文章，发布到公众号。

---

<!-- WORKFLOW_DEFINITION
{
  "id": "wf_001",
  "name": "公众号文章发布",
  "nodes": [
    {
      "id": "node_1",
      "name": "搜索资料",
      "type": "skill",
      "skill": "baidu-search",
      "input": { "query": "${user_input.topic}" }
    },
    {
      "id": "node_2",
      "name": "生成文章",
      "type": "skill",
      "skill": "article-generator",
      "input": {
        "material": "${node_1.output.results}",
        "style": "${user_input.style}"
      }
    }
  ],
  "edges": [
    { "from": "node_1", "to": "node_2" }
  ]
}
WORKFLOW_DEFINITION -->

## 步骤说明

### 第一步：搜索资料
...
```

### 6.3 目录结构

```

├── 公众号文章发布/
```

## 七、工作流编排设计

### 7.1 编排方式

**核心原则：自然语言操作，可视化只呈现**

```
┌─────────────────────────────────────────────────────────────────┐
│                          编排方式                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ❌ 传统方式：可视化拖拽                                          │
│     用户需要学习拖拽节点、连线、配置参数                           │
│     → 复杂、门槛高                                                │
│                                                                  │
│  ✅ 新方式：自然语言                                              │
│     用户用自然语言描述需求                                         │
│     编排Agent理解意图，自动生成工作流                              │
│     → 简单、门槛低                                                │
│                                                                  │
│  可视化只用于：                                                   │
│     • 展示工作流结构                                              │
│     • 显示执行进度                                                │
│     • 查看数据流向                                                │
│     （不可拖拽操作）                                              │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 7.2 编排 Agent（系统级）

**角色定义**：

| 问题 | 答案 |
|------|------|
| **谁来编排？** | 编排 Agent（系统级服务） |
| **谁来执行？** | 工作流引擎（独立执行器） |

**编排 Agent 与员工 Agent 的区别**：

| Agent | 角色 | 职责 |
|-------|------|------|
| **编排 Agent** | 系统级服务 | 理解意图，创建/修改工作流定义 |
| **员工 Agent** | 员工能力延伸 | 接收员工需求，调用工作流，汇报结果 |

**编排 Agent 职责**：

```
┌─────────────────────────────────────────────────────────────────┐
│                      编排 Agent                                  │
│                      （系统级服务）                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 理解用户意图                                                  │
│     用户: "创建公众号发布工作流"                                   │
│     → 解析：需要搜索、写作、排版、发布                             │
│                                                                  │
│  2. 生成工作流定义                                                │
│     → 创建 workflow.md 文件                                       │
│     → 写入 JSON 块（nodes, edges）                                │
│                                                                  │
│  3. 修改工作流                                                    │
│     用户: "添加数据分析步骤"                                       │
│     → 解析：在哪个位置插入什么节点                                 │
│     → 更新 JSON 块                                                │
│                                                                  │
│  4. 评估与优化                                                    │
│     用户: "搜索不够全面"                                           │
│     → 分析问题节点                                                │
│     → 调整节点配置                                                │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 7.4 自然语言命令示例

| 用户输入 | 编排 Agent 动作 |
|---------|----------------|
| "创建一个XX工作流" | 解析意图，生成完整 DAG |

## 八、节点类型设计

### 8.1 节点类型（简化）

由于人在流程外，节点类型只有两种：

| 类型 | 执行者 | 说明 |
|------|--------|------|
| `skill` | 系统调用 | 自动执行某个 Skill |
| `agent` | 指定 Agent | 调用某 Agent 执行（自动选择） |

**不需要**：审批节点、人工节点、等待节点

### 8.2 Agent 节点设计

**Agent节点配置**：

```json
{
  "id": "node_2",
  "name": "数据分析",
  "type": "agent",
  "agent_id": "auto",              // auto = 自动选择
  "required_capability": "数据分析", // 需要的能力
  "skill": "data-analyzer",         // 要执行的 Skill
  "input": { "data": "${node_1.output}" }
}
```

**多Agent执行示例**：

```json
{
  "nodes": [
    {"id": "node_1", "type": "skill", "skill": "baidu-search"},
    {"id": "node_2", "type": "agent", "required_capability": "写作"},
    {"id": "node_3", "type": "agent", "required_capability": "数据分析"},
    {"id": "node_4", "type": "skill", "skill": "wechat-publisher"}
  ]
}
```

### 9.3 执行时负载检查

**工作流引擎执行前自动检查**：

```python
class WorkflowEngine:

    async def _invoke_agent(self, node: dict, input_data: dict):
        agent_id = node.get('agent_id')

        # 1. 如果是 auto，重新选择（动态调整）
        if agent_id == 'auto':
            agent_id = await self._select_best_agent(
                node.get('required_capability'),
                node.get('skill')
            )

        # 2. 检查 Agent 状态
        agent = get_agent_profile(agent_id)

        if agent.status == 'busy':
            # 自动换其他 Agent（不询问人）
            agent_id = await self._select_alternative_agent(
                node.get('required_capability')
            )

        # 3. 更新状态
        update_agent_status(agent_id, 'busy')
        agent.current_tasks += 1

        # 4. 执行
        result = await call_agent(agent_id, input_data)

        # 5. 更新状态和统计
        update_agent_status(agent_id, 'idle')
        agent.current_tasks -= 1
        update_success_rate(agent_id, success=True)

        return result

    async def _select_best_agent(self, capability: str, skill: str) -> str:
        """自动选择最优 Agent"""
        agents = query_agents_by_capability(capability)

        # 计算综合评分
        scored = []
        for agent in agents:
            if agent.status != 'idle':
                continue
            score = (
                agent.expertise_level.get(capability, 50) * 0.5 +
                agent.success_rate * 100 * 0.3 +
                (100 - agent.current_tasks * 10) * 0.2
            )
            scored.append((agent.agent_id, score))

        # 选择最高评分
        if scored:
            return max(scored, key=lambda x: x[1])[0]

        # 没有空闲 Agent，等待
        return await self._wait_for_available_agent(capability)
```

### 9.5 选择策略对比

| 策略 | 说明 | 人工参与 |
|------|------|----------|
| **自动选择最优** | 按能力匹配度+成功率+负载综合评分 | ❌ 无 |
| **动态调整** | 执行时Agent繁忙自动换其他Agent | ❌ 无 |
| **等待可用** | 无空闲Agent时自动等待 | ❌ 无 |

**设计原则**：
- 编排时自动选择 → 只告知用户结果
- 执行时动态调整 → 无需人工干预
- 异常时自动处理 → 换Agent或等待
