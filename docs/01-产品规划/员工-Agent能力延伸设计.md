# 员工-Agent 能力延伸设计

> 状态：设计完成
> 优先级：P1
> 创建时间：2026-04-10

## 一、核心理念

**Agent 是员工的能力延伸，不是数字下属**

```
传统组织：员工 → 下属 → 执行 → 汇报 → 审批
                    ↑ 层级管理、审批链

新设计：员工 ← Agent（能力延伸）
         ↑ 员工受益，Agent赋能
         ↑ 员工只参与两端（需求+验收）
```

---

## 二、角色重新定义

| 角色 | 传统定义 | 新定义 |
|------|----------|--------|
| **员工** | 任务分配者、审批者、管理者 | **需求提出者、结果验收者、受益者** |
| **Agent** | 执行工具、下属、汇报者 | **能力延伸、自主执行者、赋能者** |
| **工作流** | 需要人推动的流程 | **自动运转，无需人工节点** |

---

## 三、交互流程

### 阶段1：需求提出

```
员工: "帮我写一篇关于AI发展趋势的公众号文章"

Agent理解意图:
- 任务类型：公众号文章
- 主题：AI发展趋势
- 需要的Skill：搜索、写作、排版、发布

Agent: "我来完成，预计需要4步：搜索→写文章→排版→发布
       请确认主题，风格偏好？"

员工: "风格轻松一点"
```

### 阶段2：自主执行（员工不参与）

```
Agent: "开始执行..."

[Agent调用工作流引擎]
[搜索资料] → [生成文章] → [排版] → [发布]
     ✓            ✓           ✓        ✓

（员工可以看到进度，但不需要参与任何环节）
```

### 阶段3：结果反馈

```
Agent: "任务完成！
       文章已发布：https://mp.weixin.qq.com/xxx
       耗时：45秒"

员工: [查看结果] → [评估质量]

员工反馈：
- "文章质量不错" → Agent记录满意
- "搜索不够全面" → Agent下次优化
```

## 四、数据库设计

### Employee 表字段

| 字段 | 类型 | 用途 |
|------|------|------|
| `agent_ids` | TEXT | 绑定关系，JSON数组 `["agent1"]` |
| `agent_config` | JSONB | 能力配置偏好 |

### agent_config 结构

```json
{
  "autonomy": "high",
  "report_style": {
    "detail_level": "summary",
    "timing": "on_complete"
  },
  "learning": {
    "remember_feedback": true,
    "auto_improve": true
  }
}
```

### 字段分工

| 字段 | 存什么 | 作用 |
|------|--------|------|
| `agent_ids` | 哪些 Agent | 绑定关系（已存在） |
| `agent_config` | 怎么服务 | 配置偏好（新增） |

## 五、自主性级别

| 级别 | 行为 |
|------|------|
| **high** | 不需确认就执行，完成后汇报，失败自动重试 |
| **medium** | 需确认后执行，实时汇报进度 |
| **low** | 每步确认，每步汇报 |

---

## 六、与人类组织的差异

| 维度 | 人类系统 | Agent系统 |
|------|----------|-----------|
| 员工角色 | 管理者、审批者 | **受益者** |
| Agent角色 | 下属 | **能力延伸** |
| 流程参与 | 员工参与审批节点 | **员工只在两端** |
| 责任模式 | 审批追责 | **结果改进** |
| 效率瓶颈 | 人工审批 | **无人工瓶颈** |
