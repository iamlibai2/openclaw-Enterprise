## 议题一：Skill 禁用功能竞态条件问题

**讨论时间**：上午 09:00 - 09:30

### 问题描述

用户反馈：在 Skills 页面快速连续点击开关，第 3 次提示错误：

```
操作失败：[Errno 111] Connect call failed ('127.0.0.1', 18789)
```

### 问题分析

经过排查发现两个问题：

**问题 1：多个后端进程冲突**
```
iamlibai    7479  python app.py  (旧进程)
iamlibai   17857  python app.py  (新进程)
```

**问题 2：端口耗尽**
- 每次请求创建新的 WebSocket 连接
- 频繁连接/断开导致端口耗尽
- 第 3 次请求时无可用端口

### 解决方案

| 问题 | 解决方案 | 文件 |
|------|----------|------|
| 多进程冲突 | 终止旧进程，确保只有一个后端运行 | 运维 |
| 端口耗尽/连接问题 | 增加连接重试次数和等待时间 | `gateway_sync.py` |
| Hash 冲突 | 添加配置更新重试机制 | `app.py` |
| 快速点击 | 前端添加 loading 状态防抖 | `Skills.vue` |

**前端防抖**：
```vue

```

### 结论

- ✅ 连接重试机制已实现（`gateway_sync.py`）

## 议题三：Gateway 速率限制发现

**讨论时间**：上午 11:00 - 11:30

### 问题发现

经过深入排查，发现 Gateway 对 `config.patch` 有速率限制：

```
[UNAVAILABLE] rate limit exceeded for config.patch; retry after 59s
```

**速率限制规则**：
- `config.patch` 每分钟最多 3 次调用
- 超过限制后需要等待约 60 秒

### 这不是 Bug

Gateway 的速率限制是**设计行为**，用于：
- 保护配置文件不被频繁修改
- 防止并发配置更新导致冲突
- 确保配置变更的稳定性

### 解决方案

| 层面 | 处理方式 |
|------|----------|
| 前端 | 已有防抖机制，防止用户快速连续点击 |
| 前端 | 改进错误提示，针对速率限制显示"操作太频繁，请稍后再试" |
| 后端 | 使用持久 WebSocket 连接，避免频繁创建/关闭连接 |

### 连接管理优化

**问题**：每次请求创建新 WebSocket 连接，导致 TIME-WAIT 累积。

**解决**：实现专用线程保持持久连接：

```python
class GatewayConnectionThread(threading.Thread):
    """专用线程保持 WebSocket 连接"""
    def run(self):
        self.loop = asyncio.new_event_loop()
        while self.running:
            # 处理请求队列，复用同一连接
            self._process_requests()
```

**效果**：
- TIME-WAIT 连接从 28 个降到 6 个
- 连接复用，减少资源消耗

### 结论

- ✅ Gateway 速率限制是设计行为，不是 Bug
- ✅ 前端防抖 + 友好错误提示
- ✅ 持久连接优化完成
- ✅ 用户在正常使用时不会触发速率限制

## 议题二：系统重复代码分析与优化讨论

**讨论时间**：上午 09:30 - 10:00

### 讨论 背景

在修复竞态条件问题时，发现代码中存在大量重复模式，Claude 提出进行代码优化。

### 重复代码统计

| 模式 | 出现次数 | 示例 |
|------|----------|------|
| `jsonify({'success': True})` | 86 次 | 成功响应 |
| `jsonify({'success': False})` | 251 次 | 失败响应 |
| `except Exception as e:` | 118 次 | 异常处理 |
| `config.get` + `hash` | 20 次 | 配置更新 |
| `ElMessage.success/error` | 150 次 | 前端消息提示 |

### 优化方案讨论

Claude 提出以下优化建议：

| 方案 | 说明 |
|------|------|
| `api_success/api_error` 辅助函数 | 统一响应格式 |
| `@handle_errors` 装饰器 | 统一异常处理 |
| 前端消息工具封装 | 统一 ElMessage 调用 |

### 用户反馈

用户提出疑问：这样处理是否合适？

### 最终决策

**决定：不做过度抽象**

理由：
1. **代码重复不一定是坏事**
   - `jsonify({'success': True, 'data': xxx})` 一眼能看懂
   - 每个地方可能有微小差异（错误码、日志、额外信息）

2. **过度抽象的代价**
   - 增加理解成本（要看封装函数的实现）
   - 降低灵活性（特殊需求要绕过封装）
   - 调试困难（错误堆栈更深）

3. **当前代码的问题已经解决**
   - 连接复用 ✅
   - 重试机制 ✅
   - 前端防抖 ✅

### 设计原则

| 原则 | 说明 |
|------|------|
| 简单直接 | 一眼能看懂，不过度抽象 |
| 解决实际问题 | 只优化有问题的部分，不为了优化而优化 |
| 灵活性 | 不同场景可以有不同的处理方式 |

### 结论

- ✅ 只保留 `_patch_config()` 函数（已实现）
- ❌ 不创建 `api_success/api_error` 辅助函数
- ❌ 不创建 `@handle_errors` 装饰器
- ❌ 不封装前端消息工具
- ✅ 创建代码规范文档记录决策
