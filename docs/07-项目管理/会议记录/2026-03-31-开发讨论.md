## 议题二：Skill 与 Agent 归属关系问题

**讨论时间**：下午 15:45 - 16:00

### 讨论内容

当前 Skills 页面的处理方式存在问题：

```python
# 当前代码：用 slug 作为 key 去重
if slug not in skills:
    skills[slug] = {...}
```

**问题**：
- 如果 Agent A 有 `excel-xlsx`，Agent B 也有 `excel-xlsx`，只会显示第一个
- `agentId` 只记录了第一个拥有该 skill 的 Agent
- 信息丢失了

### 解决方案讨论

| 方案 | 说明 | 优缺点 |
|------|------|--------|
| 1. 按 Agent 分组 | 每个 Agent 一组 skills，同名 skill 可能出现多次 | 信息完整，但列表变长 |
| 2. 记录所有 Agent | `agentIds: ['neo', 'alice']`，卡片显示"属于: Neo, Alice" | 去重但保留归属信息 |
| 3. Agent 筛选器 | 下拉选择 Agent，查看该 Agent 的 skills | 灵活，需额外操作 |
| 4. 保持去重 + 标记 | 只显示一个，详情里说明"多个 Agent 共有" | 简单，但信息不完整 |

### 结论

- ✅ **用户选择方案 1**：按 Agent 分组展示
- ✅ 用不同颜色标签区分 skill 级别（workspace/shared/bundled）
- ✅ 已实现后端 `scan_skills()` 重构，返回按 Agent 分组的数据
- ✅ 已更新前端 Skills.vue 按 Agent 分组展示

### 实现后修复

- ✅ 后端修复完成

## 议题六：远程部署架构讨论

**讨论时间**：晚上 20:00 - 21:00

### 用户提问

> 现在是不是通过读取配置文件的方式进行一些设置的管理？还有没有其他的方式？如果咱们这套程序和 OpenClaw 不是部署在同一台服务器呢？

### 问题分析

当前 Admin UI 后端直接读写 `~/.openclaw/openclaw.json` 文件，存在以下限制：
- 必须与 OpenClaw Gateway 同服务器部署
- 无法远程管理多个 OpenClaw 实例

### 探索过程

通过阅读 OpenClaw 源码，发现：

1. **OpenClaw 内置 WebUI 通过 WebSocket 与 Gateway 通信**
   - 文件：`src/gateway/server/ws-connection/message-handler.ts`
   - 协议：JSON-RPC 风格的请求/响应

2. **Gateway 已提供完整的管理 API**
   - 配置管理：`config.get`, `config.patch`, `config.apply`
   - Agent 管理：`agents.list`, `agents.create`, `agents.update`, `agents.delete`
   - 文件操作：`agents.files.list`, `agents.files.get`, `agents.files.set`

### 方案讨论

| 方案 | 说明 | 优缺点 |
|------|------|--------|
| WebSocket API | Admin UI 通过 WebSocket 调用 Gateway | ✅ 标准方式，支持远程 |
| 配置存数据库 | 配置迁移到数据库 | 需要改造 OpenClaw 启动逻辑 |
| 共享存储 | NFS/SMB 共享目录 | 需要运维配合 |
| 配置中心 | 引入配置中心服务 | 适合大规模部署 |

### 结论

- ✅ **选择 WebSocket API 方案**：改动最小，标准化程度高
- ✅ 将「直接读写文件」改为「调用 WebSocket API」
- ✅ 本地保留：用户、权限、部门、员工（数据库操作）
- ✅ 改用 WebSocket：Agent、配置、Skills、文件管理
- ⏸️ 待实现 WebSocket 客户端模块

### 模块划分

**保留本地（数据库）**：
- 认证系统、用户管理、角色权限、部门管理、员工管理、操作日志、任务统计

**改用 WebSocket**：
- Agent 管理、配置管理、配置文件管理、Skill 管理、会话记录、Gateway 控制
