## 讨论内容

### 1. 问题背景

用户反馈：定时任务执行后，Admin UI 没有展示执行结果。

**现状分析**：
- 后端通过 `chat.send` 发送任务给 Agent
- Agent 执行成功（在 OpenClaw 可见）
- 但结果没有回写到 Admin UI 数据库
- 前端无法获取执行结果

### 2. 方案讨论

#### 方案 A：事件驱动（Gateway 事件监听）

```
Admin UI 后端 ←WebSocket→ Gateway
     ↓ 收到 chat 事件
     ↓ 更新数据库
     ↓ SSE 推送给前端
```

**优点**：实时、不阻塞、支持长时间任务
**缺点**：需要维护持久连接

#### 方案 B：轮询 chat.history

```
发送任务 → 等待 60 秒 → 轮询 chat.history → 获取结果
```

**优点**：实现简单
**缺点**：阻塞调度器、超时问题

### 3. 前端直连 vs 后端监听

用户提问：为什么不能像 Chat 页面一样，让浏览器直连 Gateway？

**回答**：

| 场景 | 用户状态 | 浏览器状态 | 接收方式 |
|------|---------|-----------|---------|
| Chat 对话 | 在线等待 | 打开 | 浏览器直连 ✅ |
| 定时任务 | 可能离线 | 可能关闭 | 后端监听 ✅ |

**核心区别**：定时任务是无人值守的后台任务，用户可能不在电脑前，结果必须可靠存储到数据库。

### 4. SSE vs 轮询

#### 用户提问：轮询间隔写死不好，如何可配置？

**方案对比**：

| 方案 | 实时性 | 资源消耗 | 复杂度 |
|------|--------|---------|--------|
| HTTP 轮询 | 低 | 高（频繁请求） | 低 |
| WebSocket | 高 | 中 | 高 |
| SSE | 高 | 低（空闲零开销） | 中 |

#### 用户提问：SSE 是否增加系统压力？

**回答**：

| 指标 | SSE | 轮询（60秒） |
|------|-----|-------------|
| 连接数 | 1 个长连接 | 0 |
| 每天请求 | 有事件才推送 | 1440 次/用户 |
| 空闲流量 | 心跳 ~几字节 | 完整请求 ~几百字节 |

**结论**：事件少时 SSE 更省资源，事件多时差不多。

### 5. 最终决策

**采用方案：数据库配置 + SSE 推送**

理由：
1. 考虑后期复用性（更多事件类型）
2. 企业级实时性要求
3. 资源消耗可控
4. 配置灵活
