### 1. Clear Context 插件开发

**时间**：15:00 - 15:58

**背景**：用户需要一个插件，在飞书对话中执行 `/clear-context` 命令后，清空当前会话的上下文和系统提示词，让模型以全新状态响应后续消息。

#### 1.1 技术方案选择

经过调研 OpenClaw 插件 SDK，选择方案：

```
方案: registerCommand + before_prompt_build 钩子
├── registerCommand → 注册命令，跳过 LLM，直接响应
├── before_prompt_build → 修改 prompt 相关内容
└── 不需要独占 slot → 可以与其他插件共存
```

#### 1.2 插件实现

**文件位置**：`~/.openclaw/extensions/clear-context/`

**核心功能**：

1. **命令注册**：
   - `/clear-context`：设置 `cleared=true` 状态
   - `/restore-context`：恢复正常状态

2. **状态管理**：
   - 状态持久化：`~/.openclaw/clear-context-state.json`
   - 会话隔离：不同用户独立状态
   - 用户标识提取：从 sessionKey 提取 `ou_xxx` 作为用户标识

3. **钩子实现**：

```typescript
// before_prompt_build 钩子
api.on("before_prompt_build", (event, ctx) => {
  if (state[userKey]?.cleared) {
    return {
      systemPrompt: "你是一个助手。请简洁地回答用户问题。",
      prependContext: "",  // 置空，不追加提示
    };
  }
  return {};
});

// llm_input 钩子（调试）
api.on("llm_input", (event, ctx) => {
  // 记录完整信息到日志文件
  // 用于验证实际发送给模型的内容
});
```

4. **Gateway API**：
   - `clearContext.status`：查询状态
   - `clearContext.setCleared`：设置状态

#### 1.3 问题排查过程

##### 问题 1: systemPrompt 空字符串不生效

```
现象: 返回 systemPrompt: "" 无效果
排查: 阅读 OpenClaw 源码
原因: 空字符串被视为"不修改"而非"清空"
解决: 返回非空字符串进行替换
```

##### 问题 2: ContextEngine slot 覆盖 systemPrompt

```
现象: 返回 systemPrompt: "新内容" 后仍使用原提示词
排查: 检查 openclaw.json 配置
原因: "contextEngine": "focus-clear" 占据 slot，覆盖了返回值
解决: 移除 plugins.slots 中的 contextEngine 配置
```

##### 问题 3: Gateway 未加载新代码

```
现象: 修改插件后功能不变
排查: 对比 Gateway 启动时间和插件编译时间
原因: Gateway 进程未重启
解决: pkill -9 -f "openclaw-gateway" 强制重启
```

## 技术要点

### 1. systemPrompt 替换规则

```typescript
// ❌ 不生效
return { systemPrompt: "" };

// ✅ 可以替换
return { systemPrompt: "新的系统提示词" };
```

**原因**：OpenClaw 内部实现使用 `trim()` 判断：
```javascript
const legacySystemPrompt = hookResult?.systemPrompt?.trim() || "";
if (legacySystemPrompt) {  // 只有非空才应用
  applySystemPromptOverride(session, legacySystemPrompt);
}
```

### 2. ContextEngine slot 影响

```
配置 contextEngine slot 后:
├── ContextEngine 完全接管消息处理流程
├── before_prompt_build 的 systemPrompt 返回值被覆盖
└── 只能通过 systemPromptAddition 追加内容，不能替换
```

**解决方案**：移除 slot 配置后，systemPrompt 替换正常生效。

### 3. 钩子执行顺序

```
用户消息
    │
    ▼
inbound_claim      ← 最早，可拦截
    │
    ▼
before_agent_start ← Agent 启动前
    │
    ▼
before_prompt_build ← 构建 prompt 前（可修改 systemPrompt）
    │
    ▼
llm_input          ← 发送给 LLM 前（只读，可验证）
    │
    ▼
[LLM]              ← 模型推理
```

### 4. `/context detail` 与实际内容的区别

| 命令/钩子 | 测量内容 | 是否经过钩子 |
|-----------|----------|--------------|
| `/context detail` | 配置状态（Agent 原始配置） | ❌ 不经过 |
| `llm_input` | 实际发送给模型的内容 | ✅ 经过钩子处理 |

**结论**：`/context detail` 显示的是"配置文件里有什么"，不是"模型实际收到什么"。

### 5. 消息内容追加

```typescript
// prependContext: 追加到用户消息前
return { prependContext: "这是前置上下文\n\n" };

// prependSystemContext: 追加到系统提示词前（可被缓存）
return { prependSystemContext: "[插件提示]\n\n" };

// appendSystemContext: 追加到系统提示词后
return { appendSystemContext: "\n\n[插件提示]" };
```

## 关键代码片段

### 插件主结构

```typescript
import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";

const state: PluginState = {};
const SIMPLE_SYSTEM_PROMPT = `你是一个助手。请简洁地回答用户问题。`;
const CLEAR_PROMPT = "";  // 置空，不追加提示

export default definePluginEntry({
  id: "clear-context",
  name: "Clear Context",

  register(api) {
    // 加载持久化状态
    Object.assign(state, loadState());

    // 注册命令
    api.registerCommand({
      name: "clear-context",
      handler: (ctx) => {
        state[userKey] = { cleared: true, clearedAt: new Date().toISOString() };
        saveState(state);
        return { type: "text", text: "已清空上下文。" };
      },
    });

    // 注册钩子
    api.on("before_prompt_build", (event, ctx) => {
      if (state[userKey]?.cleared) {
        return {
          systemPrompt: SIMPLE_SYSTEM_PROMPT,
          prependContext: CLEAR_PROMPT,
        };
      }
      return {};
    });
  },
});
```

### llm_input 调试钩子

api.on("llm_input", (event, ctx) => {
